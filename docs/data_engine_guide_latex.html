Logo Data Engine 使用說明與資料流規格（LaTeX 風格版）

版本：v1.1（對應 /raid/ming/logo/logo_data_engine）

1. 系統概述

本系統為一個可重跑、可擴充、可審核的 Logo 資料引擎。核心原則為： 所有資料都進入同一 schema（ImageRecord → LogoInstance → BrandRecord）， 並以 tier 作為唯一狀態機。

Pipeline 預設包含：Collector、Ontology、Ingest、Quality Gate、Segmentation（SAM3）、 Annotation（OCR/CLIP/Caption/VLM）、Review、Export。

2. 環境與啟動

2.1 安裝與更新

cd /raid/ming/logo
./logo_data_engine/setup_env.sh logo_sam3

此腳本會：

  • 建立 conda 環境（Python 3.12）
  • 安裝依賴、Torch、PaddleOCR、Ultralytics、HuggingFace 模型
  • 下載或更新 SAM3 至 /raid/ming/logo/logo_data_engine/sam3

2.2 快速執行（多品牌 batch）

./logo_data_engine/run_multibrand_batch.sh \
  --run-name batch_10 \
  --target-records 10 \
  --with-sam3 \
  --use-vlm \
  --vlm-model-id llava-hf/llava-1.5-7b-hf

2.3 GPU 指定

./logo_data_engine/run_multibrand_batch.sh \
  --run-name batch_10 \
  --target-records 10 \
  --with-sam3 \
  --cuda-devices 1,2,3

3. Pipeline 預設流程與觸發

每次執行會先跑 preflight，輸出：

results/<run_name>/meta/preflight.json

內容包含所有模型的可用性與錯誤訊息。

6. 模型 Prompts 與輸出樣例（詳細）

6.1 GroundingDINO（Logo bbox）

Prompt 來源（由 build_prompts 組合）：

  • 品牌詞：{brand} logo, {brand} brand logo, {brand} emblem
  • Alias：{alias} logo
  • 物件結合：{brand} logo on {object}, {object} logo
  • 歷史語境：vintage {brand} logo, retro {brand} logo

範例 Prompt 列表：

nike logo
nike brand logo
nike emblem
shoe logo
nike logo on shoe
vintage nike logo

輸出樣例（節錄）：

{
  "grounding_bbox_xyxy": [112.3, 76.1, 188.4, 132.7],
  "grounding_score": 0.71,
  "grounding_label": "nike logo"
}

6.2 GroundingDINO（Object-first 物件框）

Prompt 來源（依 category 與 object_terms）：

  • shoes：shoe, sneaker, footwear
  • apparel：shirt, hoodie, apparel
  • smartphones：phone back, mobile phone
  • 通用：product, object, item

輸出樣例：

{
  "object_grounding_bbox_xyxy": [18.5, 44.2, 502.9, 363.7],
  "object_grounding_label": "sneaker",
  "object_grounding_score": 0.83
}

6.3 SAM3 Segmentation

輸出欄位：

  • logo_bbox_xyxy, logo_mask_path
  • logo_mask_area_pixels, logo_mask_area_fraction
  • logo_segmentation_status

輸出樣例：

{
  "logo_bbox_xyxy": [120.0, 78.2, 182.4, 124.9],
  "logo_mask_path": ".../segment/masks/record_001.png",
  "logo_mask_area_pixels": 2412,
  "logo_mask_area_fraction": 0.0032
}

6.4 PaddleOCR

輸出樣例：

{
  "ocr_text": "NIKE",
  "ocr_confidence": 0.92,
  "ocr_lines": [
    {"text": "NIKE", "score": 0.92, "polygon": [[10,10],[80,10],[80,30],[10,30]]}
  ]
}

6.5 CLIP Logo Gate

Prompt：

  • a photo with a brand logo

輸出樣例（quality_gate_json 內）：

{
  "clip_logo": {
    "score": 0.27,
    "threshold": 0.22,
    "soft_floor": 0.08,
    "decision": "passed"
  }
}

6.6 CLIP Brand Retrieval

輸出樣例：

{
  "clip_score": 0.86,
  "top_matches": [
    {"brand": "nike", "score": 0.86},
    {"brand": "adidas", "score": 0.63},
    {"brand": "new balance", "score": 0.57}

6.7 BLIP Caption

Prompt：

  • Describe the brand, object, and photo scene.
  • Describe the logo, brand, and object in this crop.

輸出樣例：

{
  "full_image_caption": "A person holding a red Coca-Cola can.",
  "logo_crop_caption": "The Coca-Cola logo on a soda can."
}

6.8 VLM (LLaVA)

Prompt：

  • Describe the brand, object, and photo scene.
  • Describe the logo, brand, and object in this crop.

輸出樣例：

{
  "full_image_caption": "A Nike sneaker displayed on a table.",
  "logo_crop_caption": "A white Nike swoosh on the shoe."
}

7. Quality Filter 決策規則（更細）

每張圖依序過五關，任一關失敗即標記 quality_status=filtered。

  • 若來源可信且 CLIP score ≥ soft floor，會進入 soft pass（仍保留失敗原因）。
  • pHash 去重優先於昂貴模型，避免重複算力浪費。
  • 寫回欄位：quality_status, quality_score, quality_gate_json

8. SAM3 Two-Stage 流程

預設 object-first：

  1. GroundingDINO object prompts → 物件 bbox
  2. 對每個物件 crop → GroundingDINO logo prompts
  3. 使用 logo bbox + SAM3 → mask

輸出欄位（segment records）：

  • object_grounding_bbox_xyxy
  • object_grounding_label
  • logo_grounding_bbox_xyxy
  • logo_mask_path

9. OCR / VLM / BLIP / Qwen 輸出位置

所有 OCR、Caption、Qwen 推理與 Attribution 結果都會寫入： DB.logo_instances，並輸出到 export/knowledge_base.json。

主要欄位：

  • ocr_text, ocr_confidence
  • caption_text, caption_model
  • attribution_json（包含 clip_retrieval / captioning / qwen_knowledge / merged）
  • knowledge_json（Qwen 多段式知識推理）
  • risk_json（Qwen 風險判斷）

完整原始 payload：logo_instances.provenance_json

注意：clip_retrieval 現在預設只保留為診斷訊號， 不再直接主導最終品牌合併；最終品牌主要由 trusted metadata、OCR、caption / VLM 與 fallback similarity 決定。

9.1 Qwen 多段式推理

Qwen 不再一次問完整大題，而是拆成五段各自輸出 JSON：

  1. grounding
  2. recognition
  3. world_knowledge
  4. risk
  5. ambiguity

好處是某一段失敗時，不會讓整個 knowledge/risk 全部變成 null。

最終寫回：

  • logo_instances.knowledge_json
  • logo_instances.risk_json
  • logo_instances.attribution_json.qwen_validation

11. Knowledge Base 內容與結構（更詳細）

輸出檔：export/knowledge_base.json

結構為「每張影像一筆」，內含 logo_instances 與 brand_record：

{
  "image_id": "...",
  "engine_quality_status": "passed",
  "engine_quality_gate": {...},
  "engine_brand_record": {...},
  "logo_instances": [
    {"instance_id": "...", "bbox": [...], "attribution": {...}, "brand_record": {...}}
  ]

內容結構：

  • ImageRecord：影像與來源資訊
  • LogoInstance：bbox / mask / OCR / attribution 結果
  • BrandRecord：品牌知識（industry / country / aliases）

核心欄位：

LogoInstance bbox_json, mask_path, ocr_text, clip_score, caption_text, attribution_json, knowledge_json, risk_json, tier

14.5 欄位驗證實作規則（正規化/允許值/檢查邏輯）

14.5.1 正規化邏輯

  • brand_id：使用 slugify（小寫 + 非英數替換成 -）
  • source_channel/source_kind/category：strip().lower()
  • scene_bucket：優先 capture_context.scene_type，否則 source_channel，再否則 category
  • image_id：stable_hash(external_key, brand, category)
  • instance_id：stable_hash(image_id, proposal)

14.5.2 允許值列表（目前 pipeline 使用）

欄位允許值/建議值說明
tierproposal / silver / gold固定三態
review_statusauto_accept / needs_review目前 pipeline 使用
quality_statuspassed / filtered / pending / errorquality gate 輸出
categoryshoes / apparel / mugs / cans / smartphones內建 prompts 支援
source_kindofficial / marketplace / othertrusted catalog 判斷
source_channelecommerce / marketplace / advertising / editorial / social / sports_broadcast / user_capture非硬性限制，主要用於 scene_bucket

14.5.3 Gate 驗證邏輯（實作細節）

  • 尺寸：width >= 256、height >= 256
  • 格式：JPEG/PNG/WEBP/JPG
  • 模糊：Laplacian variance >= 45.0
  • pHash：distance <= 6
  • CLIP logo score：score >= 0.22；trusted catalog 可 soft pass (0.08)
  • YOLO prescreen：至少一個 bbox conf >= 0.3

14.5.4 BBox / Mask 基本驗證

  • bbox：長度=4 且 x0 < x1, y0 < y1
  • mask_area_fraction：0 ~ 1

14.5.5 正規化範例與合法/非法例子

欄位正規化規則合法例子非法例子
brand_id slugify（小寫+非英數轉 -） 输入: "New Balance" → new-balance 空字串 → unknown
source_channel strip + lower "Advertising " → advertising null/空字串
source_kind strip + lower "Marketplace" → marketplace 非允許值（如 "foo")
category strip + lower "Shoes" → shoes 不在清單（如 "car")
scene_bucket scene_type > source_channel > category scene_type="stadium" → stadium 全部缺失 → unknown
image_id stable_hash(external_key, brand, category) 固定輸入 → 固定輸出 brand/category 變動 → 新 id
bbox [x0,y0,x1,y1] 且 x0<x1,y0<y1 [10,20,80,60] [80,20,10,60]
mask_area_fraction 0 ~ 1 0.0032 -0.2 或 1.5

14.5.6 錯誤訊息範本與補救策略

欄位錯誤訊息範本補救策略
brand_id brand_id_invalid: empty or unknown 回填 brand 原字串再做 slugify；若仍空則設 unknown
category category_invalid: not in allowed list 映射到最近類別（如 "sneakers"→"shoes"），或保留原值並標記 difficulty_flags
source_channel source_channel_missing 由 source_kind 或 source_name 推測，或設為 "unknown"
image_url image_url_invalid: not a URL 使用 local_image_path，或標記 missing_image
bbox bbox_invalid: x0>=x1 or y0>=y1 回退到 GroundingDINO 次佳候選框，或標記 needs_review
mask_area_fraction mask_fraction_invalid: out of range 視為 mask 不可信，保留 bbox 但標記 logo_mask_review_status=poor
quality_status quality_gate_failed: {reason} 不進入 annotate；保留於 filtered 以供重跑調整閾值

14.5.7 驗證實作規則（對應程式邏輯）

  • slugify：re.sub(r"[^A-Za-z0-9._-]+","-", value.lower())
  • brand_id：canonical_brand_id = slugify(name)
  • image_external_key：使用 record_id/brand/category/product_url/image_url 等組合做 stable_hash
  • image_id：stable_hash({external_key, brand, category})
  • instance_id：stable_hash({image_id, proposal})
  • scene_bucket：scene_type > source_channel > category
  • trusted catalog：source_kind in {official, marketplace} 或 source_channel in {ecommerce, marketplace}
  • tier：允許值 proposal/silver/gold（TIER_VALUES）

16. 爬蟲方案與流程

16.1 Collector 策略來源

  • 商品型來源：品牌官網 / 商品頁（如 Adidas、Lululemon）
  • 圖片型來源：Unsplash、Flickr（API 友好）
  • 資料集來源：LAION / CSV index 篩選
  • 外部野外圖：手拍 / 社群 / sports / advertising（manifest 或 folder）

16.2 Collector 實際流程

  1. 依照 brand × category 配對（或 manifest 清單）
  2. 呼叫對應 collector（產品、外部資料、manifest）
  3. 輸出 raw_records.json + 下載圖片
  4. 進入 balance/ontology/ingest pipeline

16.3 Rate Limit 與重試

  • 各來源具有不同限制，collector 內含 retry
  • 若遇 403（如 Lululemon/Adidas），會 fallback 到下一策略

16.4 品牌/網頁候選名單位置與決策規則

候選名單定義於：

/raid/ming/logo/multi_brand_fetcher.py → BRAND_SOURCES

結構為：(brand, category) → [strategy1, strategy2, ...]

決策流程：

  1. --all：直接取 BRAND_SOURCES.keys()
  2. --brands + --categories：只保留存在於 BRAND_SOURCES 的 pair
  3. 每個 pair 依策略順序嘗試；只要某策略回傳 ≥ min_records（預設 3）就停止

策略類型範例：

  • next_data：抓 Next.js JSON
  • magento_bs4：抓 Magento HTML
  • jsonld_per_page：抓 JSON-LD
  • poshmark：搜尋 fallback

17. 常見問題

17.1 SAM3 無法載入

確認：

/raid/ming/logo/logo_data_engine/sam3/sam3/model_builder.py

並確認 meta/preflight.json 內 SAM3 為 available。

17.2 VLM OOM

若 VLM 出現 CUDA OOM，請限制 GPU 或改用小模型。
